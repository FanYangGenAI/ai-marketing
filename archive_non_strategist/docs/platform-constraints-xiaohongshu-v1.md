# 平台通用硬约束方案（小红书）— 初稿 v1

> 状态：初稿  
> 日期：2026-03-23  
> 关联：`src/config/platforms/xiaohongshu.json`、`docs/system-design-draft.md`、`docs/cold-start-ingestion-v1.md`

---

## 1. 背景与问题

当前在小红书图文产出中，出现了高频合规问题：

- 缺少标题
- 标题超出平台限制（> 20 字）
- 正文字数超过平台限制（> 1000 字）

这类问题属于**平台硬约束**，不应受用户偏好影响。系统需要将“平台要求”与“用户偏好”分层管理，并在 Agent 流程中强制执行。

---

## 2. 目标

1. 建立平台侧**通用硬约束**（先覆盖小红书）。
2. 在前端提供**只读可见**的规则展示，帮助用户理解系统约束。
3. 在后端 Agent 工作流中注入并强校验，避免生成不合规文案。
4. 形成“生成 → 校验 → 回修”的闭环，而不是只靠 prompt 提醒。

---

## 3. 约束分层模型

### 3.1 Platform Hard Rules（平台硬约束）

- 来源：平台规则
- 特性：不可被用户修改或覆盖
- 执行：必须通过程序化校验器判定

### 3.2 Project Preferences（项目偏好）

- 来源：用户配置
- 特性：可编辑
- 执行：仅在不违反硬约束时生效

> 优先级：`平台硬约束 > 用户偏好 > Agent 创意自由度`

---

## 4. 小红书规则集（v1 建议）

基于当前痛点，先上线最小闭环规则：

### 4.1 强制字段与长度

- `title.required = true`
- `title.max_chars = 20`
- `body.required = true`
- `body.max_chars = 1000`

### 4.2 与现有平台配置的关系

仓库已存在 `src/config/platforms/xiaohongshu.json`，其中已有：

- `title_max_chars = 20`
- `body_chars.max = 1000`

本方案建议在此基础上补齐“required”等字段，并将规则分为：

- `hard_rules`（程序强校验）
- `guidelines`（建议项，不阻断）

### 4.3 后续扩展（v2+）

- 标签数量约束（如 min/max）
- 违禁表达检测（基于词表/规则）
- 结构要求（如首段钩子、末段 CTA）

---

## 5. 前端展示方案（只读）

### 5.1 页面位置

可放在以下任一位置：

- `ProductSettings` 增加“平台要求（只读）”卡片（推荐，用户最易发现）
- 或新增独立 `Platform Rules` 视图

### 5.2 展示字段

- 平台名（小红书）
- 规则版本、更新时间
- 硬约束列表（明显标记“不可修改”）
- 建议项列表（标记“建议”）

### 5.3 用户体验目标

- 让用户明确知道：为什么系统会回修标题/字数
- 降低“系统行为不可解释”的摩擦

---

## 6. 后端 Agent 注入与执行闭环

### 6.1 注入点（建议）

- `Planner`：在策划阶段知晓平台限制，避免不合理方向
- `Scriptwriter`：产出结构化文案时直接遵守硬约束
- `Creator`：最终组包前再次检查
- `Audit`：出具可解释错误原因

### 6.2 执行机制

1. Agent 产出文案（含 `title`、`body`）。
2. 统一校验器执行硬规则检查（非 LLM，确定性）。
3. 若失败，生成标准化错误码并路由 Reviser 定向回修。
4. 通过后进入下一阶段。

### 6.3 错误码建议

- `TITLE_MISSING`
- `TITLE_TOO_LONG`
- `BODY_MISSING`
- `BODY_TOO_LONG`

错误码用于：

- 前端可解释反馈
- Reviser 精准修复（避免整稿重写）

---

## 7. 数据结构建议（概念）

建议在平台配置中引入统一结构（示意）：

```json
{
  "platform": "xiaohongshu",
  "version": "1.0",
  "hard_rules": {
    "title": { "required": true, "max_chars": 20 },
    "body": { "required": true, "max_chars": 1000 }
  },
  "guidelines": {
    "hashtags": { "min": 3, "max": 8 },
    "tone": "口语化、真实感、生活化"
  }
}
```

> 注：此处仅为方案结构草图，便于后续实现统一解析与多平台扩展。

---

## 8. 分阶段落地建议

### Phase 1（立即止血）

- 小红书硬约束仅覆盖标题/正文必填与长度上限
- 前端只读展示
- 后端统一校验 + 回修闭环

### Phase 2（稳态增强）

- 标签/违禁词规则
- 多平台规则并行（抖音/TikTok）
- 规则版本化与审计日志

---

## 9. 验收标准（DoD）

1. 前端可见小红书平台规则（只读）。
2. 任何发布包都不会出现：无标题、标题 > 20、正文 > 1000。
3. 校验失败可返回明确错误码，并触发回修。
4. 对同一输入，多次运行校验结果一致（确定性）。
